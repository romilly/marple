## Arc 1 — Test tree reorganisation

### Motivation

Two problems drove the work:

1. Tests for any given primitive / construct / service were smeared across
   many files — scalar addition was covered in `tests/test_functions.py`
   (APLArray method level), `tests/new_engine/test_interpreter.py` (via the
   Interpreter), and `tests/new_engine/test_integration.py` (parser + executor)
   with no obvious entry point.
2. `tests/new_engine/` was a legacy name — the "new engine" has been the
   only engine for many versions.

### Classification: GOOS (not the ad-hoc rule I proposed first)

Initial framing used a shallow rule — "uses `interp.run()` → integration."
User pushed back: under that rule almost everything was integration and the
unit tier was nearly empty. The real boundary is:

- **Unit**: exercises our code and only our code. Collaborators may be real
  (sociable unit test) or doubled (mocks, fakes). Tests that call `interp.run()`
  are unit tests of the Interpreter. Tests that use a `Fake*` adapter are
  still unit — fakes are test doubles, not real external systems.
- **Integration**: our code meets code we can't change. Real adapter
  against real external thing. For MARPLE: `test_websocket.py` uses a real
  aiohttp server via `aiohttp_client` — that's the only genuine integration
  test at the moment.
- **E2E / acceptance**: whole-system user scenarios. Real server, real
  browser, real subprocess. The Playwright phase-tests and the PRIDE E2E
  against a real `marple-server` subprocess.

CLAUDE.md and the plan both updated to state this classification explicitly.
See commit `5a2036d` (doc) and `dd2b152` / `91cbf5c` (moving two earlier
miscommits from `integration/` to `unit/`).

## Arc 2 — APLArray inheritance refactor (first slice)

### Motivation

MARPLE's numpy lock-in killed the Pico path. Previous Pico work ran on
ulab.numpy but was abandoned at v0.8.0 because char data migrated to
`uint32`, which ulab doesn't support. The planned `uint32 → uint16`
workaround was deferred and never landed.

The refactor introduces a subclass seam on the core value type so a future
`UlabAPLArray(APLArray)` can plug in its own numeric machinery without
touching any APL semantics.

### Architecture chosen: inheritance, not composition

Originally I proposed a separate `NumericBackend` port with APLArray
delegating to it. User preferred the cleaner inheritance model: APLArray
becomes (eventually) an ABC with backend-specific concrete subclasses.
`NumpyAPLArray` today; `UlabAPLArray` later.

### Baby-steps refactor (incremental, not big-bang)

Safety net: tag `pre-aplarray-port-v0.8.76` pushed. Six planned phases
(A–F); this session landed A through D plus executable verification. E
(actually marking methods `@abstractmethod`) and F (test migration) were
deferred — deliberately. APLArray stays concrete in this slice; subclass
authors need only override what they want to change.

**Phase A** — subclass scaffold. `NumpyAPLArray(APLArray)` as empty subclass.
Factory on `Interpreter`: `array_cls` parameter + `make_array(shape, data)`
helper.

**Phase B** — internal construction threads the subclass. `APLArray.array`
and `APLArray.scalar` now honour `cls` (previously hardcoded
`APLArray(...)`). All internal construction inside `numpy_array.py` changed
to `type(self).array(...)` / `type(self).scalar(...)` / `type(self)(...)`.
This was extended beyond the 5 target ops to every internal construction
— otherwise subclass propagation would be partial.

**Phase C** — external construction sites migrate. Every
`APLArray(...)` / `APLArray.array(...)` call in `src/marple/` (executor,
structural, cells, environment, operator_binding, fmt, formatting,
system_commands, workspace, jupyter/html_render, stdlib/str_impl) now
constructs `NumpyAPLArray` explicitly. `S()` was redirected to
`NumpyAPLArray.scalar` — a single-line change propagating through
dozens of call sites.

### Option 1 vs Option 2 for Phase D

Phase D was originally planned as mechanical per-op extraction:
`_primitive_add`, `_primitive_subtract`, `_primitive_multiply`,
`_primitive_negate`, `_primitive_reciprocal`. Reading the actual code
revealed this was cosmetic for the dyadic ops — they all already funnel
through `_numeric_dyadic_op`, which is the real backend hook (it owns the
`np.errstate` overflow trap and `maybe_upcast`).

Switched to Option 2: extract the *real* hooks, not one-per-op ceremony.

- `_numeric_dyadic_op`: documented as the backend override hook for
  dyadic numeric arithmetic. No code change.
- `_primitive_negate`: extracted from `negate()`. Public method keeps the
  char-rejection guard; the primitive does the unary-minus work.
- `_primitive_reciprocal`: extracted from `reciprocal()`. Contract: must
  raise `DomainError("Division by zero")` on zero operand.

Three override points — one per axis where backend machinery actually
differs — rather than five trivial indirections.

### Verification

Ten new tests in `tests/unit/array/test_arraymodel.py`:

- `TestSubclassPropagation` (7 tests): factories and first-slice ops on a
  `NumpyAPLArray` instance produce `NumpyAPLArray` results, not plain
  `APLArray`.
- `TestBackendOverridability` (3 tests): subclass overrides of each hook
  are called through the public API. Doubles as executable documentation
  of the extension pattern UlabAPLArray will follow.

Test count: 1351 → 1361. `pyright src/` and `pyright tests/` clean.

### Noted for future work

- **Phase E / F** (making APLArray abstract and migrating test-side direct
  constructors) deferred until a second backend actually exists. No point
  abstracting a class with one subclass; the test-migration cost is high
  (hundreds of `APLArray.array([3], [1,2,3])` sites in tests) and only
  Option 2's clean seams are needed for Pico enablement.
- **Extending hooks to reduce / scan / dot / outer product.** The
  `np.errstate` pattern is duplicated in `operator_binding.py` and
  `executor.py`. Those sites need to route through the backend hook too
  before a UlabAPLArray can fully replace numpy. Second slice.
- **Char dtype abstraction.** `is_char_array` in `backend_functions.py`
  still hardcodes `uint32`. Separate slice; blocker for Pico (which needs
  `uint16` chars).
- **Test assertion sites using `APLArray.array(...)`**. Currently fine —
  `==` is shape + data, class-agnostic. If and when Phase E happens, these
  will need mechanical migration.

## Lessons logged

- **Module-level vs per-Interpreter injection for computational services**:
  debated at length, settled on per-Interpreter via `array_cls` parameter
  to match the existing port convention (Timer, Console, Config, FileSystem).
  Module-level had a simpler signature story but drift from the convention
  wasn't worth the saving.
- **"Composition vs inheritance" framing**: I initially assumed composition
  (NumericBackend port + delegation). User steered to inheritance early and
  correctly — less ceremony, one fewer class, and the pattern matches the
  `Filesystem / OsFilesystem / FakeFilesystem` idiom already in place for
  I/O ports, just applied at the value-type level instead of the I/O
  boundary.
- **Option 2 over Option 1 for Phase D**: mechanical per-op extraction
  proves a pattern but adds indirection that matches no real axis of
  variation. Extracting the actual backend hook (one place per category)
  is more honest.
- **GOOS classification**: every time someone asks "is this a unit test or
  an integration test?" the answer is "what boundary is being tested?"
  not "does it call more than one of our functions?"
