## Arc 1 — The failed Pico migration (6 commits, later reverted)

The morning's plan was to land the long-deferred char-dtype
migration from uint32 to uint16 so the Pico could finally run
MARPLE again. The Pico had been blocked since 2026-04-07 on
`AttributeError: module 'ulab.numpy' has no attribute 'uint32'`.

Six commits landed on main, all desktop-green:

| commit | what |
|---|---|
| `dc52bdf` | `CHAR_DTYPE` constant introduced (`np.uint32` desktop, `np.uint16` ulab) |
| `235f58e` | Dyadic ⎕DR accepts both 160 and 320 as char target code |
| `d2f0f5c` | Remaining empty-char-array sites routed through `CHAR_DTYPE` |
| `086d5b9` | Dropped `terminal.py` from the Pico deploy list (latent ImportError) |
| `03e7c89` | Conditional scalar storage: 0-d desktop, 1-d length-1 Pico |
| `4e441e0` | `scalar_item` helper replaces ~30 `.data.item()` call sites |

The scalar-storage fix was the big one. Direct mpremote probes on
the connected Pico revealed ulab's quirks one at a time:

- **No 0-dimensional arrays.** `np.asarray(7)` returns the bare
  Python int, not a 0-d ndarray. `reshape(())` silently produces
  an *empty* 1-d array, not a rank-0 scalar. The v0.7.40 scalar
  convention invariant ("data.shape matches APL shape") therefore
  cannot hold on ulab — APL scalars have to be stored as 1-d
  length-1 on that platform while the APL shape stays `[]`.
- **`.item()` doesn't exist on ulab ndarrays.** `hasattr(ndarray,
  'item') == False`. Every `value.data.item()` call crashes with
  AttributeError. Fixed by introducing a `scalar_item(x)` helper
  that falls back to `data[0]` when `.item` is missing, then
  sweeping ~30 call sites across 12 files to use it.
- **ulab upcasts integer literals to float32.** `np.array([7])`
  produces `array([7.0], dtype=float32)`. Accepted reluctantly
  — would have bitten us later for large integers.
- **`np.uint16` is an `int`, not a callable dtype.** `type(np.uint16)
  == int; np.uint16 == 72`. Calling `np.uint16(32)` raises `'int'
  object isn't callable`. Fixed `char_fill` to return a plain
  codepoint rather than a typed numpy scalar.

At that point **43 of 54** Pico tests passed. The remaining 10
broke on three further ulab divergences:

- **`np.transpose` missing** (1 test) — used by monadic ⍉.
- **`np.repeat` missing** (6 tests) — used by replicate/compress
  (`/`, `⌿`).
- **Broken fancy index assignment** (3 tests) — used by
  expand (`\`). 2-D `arr[..., list]` raises "too many indices";
  1-D fancy index assignment silently no-ops.

Fixes for all three were written and tested desktop-green but
never committed.

## The decision

At that point the user called it. The cumulative list of ulab
divergences had grown to:

- no 0-d arrays
- no `.item()`
- no `np.transpose`
- no `np.repeat`
- no `np.atleast_1d` (dyadic transpose would fail next)
- no `np.indices`
- no `np.broadcast_to` / `broadcast_arrays`
- no `np.full_like`
- float32 upcast on integer literals
- `np.uint16` as a plain int rather than a callable dtype
- 2-D ellipsis+list fancy indexing rejected
- 1-D fancy index assignment silently broken
- `reshape(())` producing empty arrays instead of 0-d

Maintaining a single `APLArray` that worked correctly on desktop
numpy and ulab had become untenable. Each workaround added
conditional complexity to a codebase whose desktop story was
already non-trivial, and the user's own use case for the Pico
was limited — the stronger need is for desktop processing of
mathematical PDFs where full numpy is appropriate.

**The decision: drop MicroPython entirely.** Pico and Presto
both. PRIDE (web, runs on desktop Python) stays alive.
Ports-and-adapters scaffolding stays as a test seam.

## Observations

### Why the Pico target proved untenable

The scale of ulab's divergence from numpy was larger than the
2026-04-07 plan assumed. The uint32 blocker was the first
symptom anyone had seen, but under it lay a whole stratum of
missing or differently-behaving operations. Each fix surfaced
another failure. The direct-on-hardware mpremote probes were
invaluable for getting ground truth quickly — without them the
session would have drifted through multiple deploy cycles per
issue.

The right call in hindsight would have been to smoke-test the
Pico against the current tree *before* starting the uint32
migration, so the full scope was visible from the start. We
tried, but the Pico couldn't even `import marple` until uint32
was fixed, so the smoke test was structurally unavailable.
The earliest point at which "this is worse than we thought"
became clear was after the scalar-storage fix, when arithmetic
started working but every other primitive path failed on a
different missing numpy attribute.

### What the scalar-storage invariant actually bought us

The v0.7.40 invariant ("data shape == APL shape") was the right
call for desktop numpy but the wrong call for ulab. Desktop
numpy has a genuine footgun (0-d arithmetic silently returns
numpy scalars) that the invariant defends against. Ulab has no
0-d arrays at all, so the invariant has nothing to defend
against and nothing to enforce. **Kept** the invariant unchanged
for the desktop codebase — it addresses a real bug independent
of Pico — but the dual-branch storage convention we briefly
introduced (0-d desktop / 1-d length-1 Pico) is gone.

### Dead-weight counting

Net line change across the 9 commits: **+93 / ‑1896**. The
vast bulk of that is the Pico file deletions (many hundreds of
lines of pico_eval.py, pico_client.py, deploy.sh, pico_stubs,
etc.), but the residue sweep deleted another ~150 lines of
desktop code that existed only as workarounds. The remaining
source tree is noticeably simpler. No architectural
reorganisation was needed — the ports-and-adapters scaffolding
was already clean, every adapter actively imported.

### Pre-existing slow-tier rot

`TestDL` had been failing silently since v0.7.40 because it
indexed `result.data[0]` on a 0-d scalar. The slow tier hadn't
been run between that migration and 2026-04-11. Fixed as part
of the abandonment commit. **Lesson:** run the slow tier at
least weekly, or at the end of any migration that touches
scalar storage. Consider adding this to the project's verify
checklist.
