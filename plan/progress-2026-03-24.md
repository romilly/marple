### Numeric type system (Phases 1–4)

#### Phase 1: Immediate fixes
- **⍕ format fix** — extracted `format_num` to `src/marple/formatting.py`;
  `_format_array` now formats whole-number floats as integers (`1` not `1.0`)
- **`to_array` overflow detection** — round-trip check catches silent integer
  overflow on numpy int32 and ulab int16

#### Phase 2: Upcast/downcast infrastructure
- `maybe_upcast(data)` — converts integer arrays to float before overflow-prone ops
- `maybe_downcast(data, ct)` — converts float arrays back to int when elements are
  close to whole numbers (within ⎕CT tolerance)
- `maybe_downcast_scalar(value, ct)` — scalar variant
- `to_list` now converts numpy scalars to native Python types
- Applied to `_pervade_dyadic` (add, subtract, multiply, power), reduce, scan,
  outer product, inner product
- Module-level `_DOWNCAST_CT` avoids threading ct through every function signature

#### Phase 3: Boolean dtype
- Comparisons (`= < ≤ ≥ > ≠`) and logical operations (`~ ∧ ∨`) produce uint8 arrays
- `to_bool_array()` in backend.py
- `bool_result` parameter on `_pervade_monadic` and `_pervade_dyadic`

#### Phase 4: ⎕DR and ⎕FR
- **Monadic ⎕DR** — queries internal type (11=bool, 163=int16, 323=int32, 645=float64, 80=char)
- **Dyadic ⎕DR** — converts between types (e.g. `645 ⎕DR 42` → float)
- **⎕FR** — floating-point representation control (645=float64, 1287=decimal)
- When `⎕FR←1287`, numeric literals created as `decimal.Decimal` for exact arithmetic
- `0.1+0.2 = 0.3` is true in decimal mode
