## The migration completion (commits 1-15)

### Path A — incremental, every commit green

The migration plan from `plan-scalar-convention-fix.md` was
followed with one major adjustment. The original plan had Step 8
as "remove the `__eq__` bridge" — a small certification commit.
In practice the bridge had been silently masking 24 producer-side
bugs where primitives constructed scalars via list-wrapping

commit became the place where the **constructor invariant got
enforced** (`reshape` to match APL shape), which is structurally a

now has a single point of truth for "data shape == APL shape".

### Numpy footguns surfaced and documented

These are durable lessons captured in
`project_scalar_convention_rot.md` (the memory file, now updated
to reflect the fixed state):

1. `np.asarray(1) * math.pi` returns a numpy SCALAR, not a 0-d
   ndarray. Type is `float64`, not `ndarray`. Indexing it raises
   `IndexError: invalid index to scalar variable`. Defence:
   constructor `np.asarray` coercion at the storage boundary.
2. `list(np.array(7))` raises `TypeError: iteration over a 0-d
   array`. Same for `for x in 0d_array:`.
3. `np.array(7).tolist()` returns the bare scalar `7`, not `[7]`.
   The trap inside `to_list()`. Fix: detect `data.ndim == 0` and
   wrap.
4. `np.flip(0d_array, axis=-1)` raises `AxisError`. Scalar
   reverse is identity; just return the data.
5. `np.array_equal(np.array(7), np.array([7]))` returns `False`.
   Numpy distinguishes 0-d and 1-d (1,) even when values match.
   This is the original reason the migration needed the bridge.

### Bugs found and fixed

1. **Char-scalar workspace save/load corruption.** The dead
   `isinstance(v, str)` branch in `workspace.py:_format_value` was
   hiding a real bug — char scalars were being saved as their
   numeric codepoint (`x ← 'a'` → saved as `x ← 97` → reloaded as
   integer 97). Silent data corruption. Fixed in commit
   `4039784`. Regression guard: `test_save_and_load_char_scalar`.

2. **Tokenizer silently dropped unknown characters.** The
   tokenize loop's `else: self._advance()` fall-through silently
   discarded any character not matched by an explicit handler.
   This was the root cause of *both* the missing-zilde and
   missing-commute pains earlier in the day: the tokenizer
   dropped `⍬` / `⍨`, the parser saw an incomplete token stream,
   and confusing downstream errors emerged. Fixed in commit
   `d3375e1` — the fall-through now raises `SyntaxError_` with
   the offending character. Three new tests in
   `TestTokenizerUnknownChars`. Zero regressions in the existing
   suite, which confirms nothing was secretly relying on the
   silent drop.

### Observations from the sweep

- **The constructor reshape invariant is doing work.** Several
  rewrites (encode in particular) produce intermediate results
  whose numpy shape doesn't quite match the APL shape (e.g.,
  `(1,)` from `np.atleast_1d` paths when the APL shape is `[]`).
  The constructor silently fixes this. Without that defensive
  layer the rewrites would each need explicit reshape calls.

- **Spec verification by user prevents drift.** For decode and
  encode and dyadic transpose, the user pasted Dyalog spec text
  into the conversation, and for the trickier cases (dyadic
  transpose error categories, scalar X edge cases) actually ran
  expressions in Dyalog and shared the output. Every test set
  was anchored to ground truth before the implementation went in.
  This caught at least one error category I would have got wrong
  (RANK ERROR vs DOMAIN ERROR for dyadic transpose axis
  validation).

- **`np.indices` + tuple fancy indexing is the right tool for
  diagonal-aware transpose.** No Python loop, no per-cell
  construction, handles all the spec cases naturally including
  matrix α, unit-axis extension, and the diagonal collapse from
  duplicated values in X.

- **Two operator-style "silently dropped" bugs in one day** is
  what triggered the tokenizer hardening. The lesson generalises:
  any future operator addition will fail loudly instead of
  silently if the glyph is missing from the tokenizer.
