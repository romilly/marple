### 9. The discovery

Investigated whether the scalar branch was decorative or necessary.
Discovery: `np.array_equal(np.array([7]), np.array(7))` returns
**`False`** â€” numpy does not consider 0-d and 1-d (1,) equal even
when they hold the same value. And:

```python
>>> from marple.numpy_array import S
>>> S(7).shape
[]                  # APL says: scalar (rank 0)
>>> S(7).data
array([7])          # numpy says: 1-d, shape (1,)
>>> S(7).data.ndim
1                   # numpy says: rank 1
```

**Every scalar APLArray in the codebase carries this contradiction.**
APL shape and numpy data shape disagree about rank. The scalar branch
in my decode rewrite wasn't decorative; it was compensating for the
storage convention.

User pushed: *"A convention that says APL arrays store data as a
vector is a disaster. Are you telling me that `S(7)` stores its
numpy data as a vector?"*

Yes. Confirmed by reading `numpy_array.py:43`:

### 11. How it happened

`git log -S 'def scalar' -- src/marple/numpy_array.py` points at
`4c5e77a` (2026-04-04, "Move NumpyArray to numpy_array.py, merge ABC
into concrete APLArray") for the current location of the factory.
But the convention is older. It came from the **list-backed era**
before numpy was the backend. In that era, an APL array's data was
a Python `list`, and a scalar was naturally `[value]` â€” the cheapest
flat container for one value. Code everywhere learned to do `data[0]`
to extract scalar values.

When the migration to numpy happened, the obvious mechanical
translation was `[value]` â†’ `np.array([value])`. Every existing
`data[0]` access kept working unchanged. Nobody had to rewrite
anything. The migration **avoided** `np.array(value)` (0-d) precisely
because that would have broken every existing call site.

Three things conspired to keep the rot invisible until today:

1. **Numpy length-1 broadcasting accidentally mimics APL scalar
   extension.** `np.array([2]) + np.array([1,2,3]) = [3,4,5]`,
   exactly what APL `2 + 1 2 3` should give. Most arithmetic
   primitives produced right answers for free.
2. **Primitives that did care about rank used `is_scalar()` (the APL
   shape check), not `data.ndim`.** That check still works because
   `self.shape` is the source of truth and is set independently of
   how data is stored.
3. **No primitive in the new engine has needed a result-shape rule
   that distinguishes scalar input from 1-element-vector input** â€”
   until decode (`(ÂŻ1â†“âŤ´âŤş),(1â†“âŤ´âŤµ)`), and the rank operator coming
   next will be far worse.

It's not anyone's fault. It's a textbook case of locally reasonable
choices compounding into structural rot, made invisible by lucky
broadcasting and an `is_scalar()` API that papered over the lie.

## What we discovered: the scalar convention is broken

See the new memory `project_scalar_convention_rot.md` for the durable
record. In short:

- `APLArray.scalar(v)` stores data as **1-element 1-d ndarray** (shape `(1,)`)
- The APL `shape` attribute is `[]` (rank 0)
- **The two layers disagree about rank.** `data.ndim` lies for scalars.
- 80â€“150 src sites compensate via `data[0]` indexing or `is_scalar()`
- Numpy idioms (`broadcast_to`, `@`, `cumprod` along an axis) silently
  produce wrong shapes when handed scalar APLArrays, because they see
  a phantom length-1 axis the APL layer says shouldn't exist
- The decode fix is blocked on this. So is encode. So is transpose.
  **So is the rank operator** (next major feature, per
  `project_rank_operator.md`).

## Decision: pause sweep, fix scalars first

The decode/encode/transpose sweep is paused. Tomorrow's session is the
scalar convention fix. Plan written: `plan/plan-scalar-convention-fix.md`.

Rationale: every primitive in the sweep will either inherit the rot
or carry compensating reshape code. The rank operator after the sweep
will be actively harmful to design under the current convention. Fixing
once, before further primitive work, is cheaper than each new primitive
paying the tax.
