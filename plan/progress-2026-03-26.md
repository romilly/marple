## Where we are stuck

### Parser architecture: operator binding precedence

**The problem**: Operators must bind tighter than functions in APL. `f⍤2-a` should
parse as `(f⍤2)(-a)`, not `f⍤(2-a)`. Our recursive descent parser processes
right-to-left for functions, which means `2-a` is evaluated before `⍤` can bind `2`.

**The impact**: User-defined operators work in many cases but parentheses are needed
to resolve binding ambiguity. `(double) twice 5-3` gives 8 (wrong) instead of 17.
This means the current implementation is not correct APL.

**The solution**: Rewrite the parser to follow Iverson's parsing algorithm from the
Dictionary of APL. This is a stack-based approach with a pattern-matching window
that resolves operators before function application.

**Status**: The user is converting Iverson's Dictionary of APL to markdown so we can
implement the algorithm correctly. This will be a significant parser refactoring
with 572+ tests as regression protection.
