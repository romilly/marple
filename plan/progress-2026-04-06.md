## What we're stuck on

### Problem 1: The chicken-and-egg problem

Changing `to_array` to produce uint32 for character data breaks ~76 tests simultaneously. The breakage spans creation sites, structural operations, formatting/display, and test assertions. We cannot fix them one at a time because:
- Changing one creation site to produce uint32 breaks all downstream code that consumes that data (formatting, comparisons, structural ops still expect list[str])
- But we can't fix the consumers first because they currently receive list[str]

### Problem 2: Empty character arrays

`to_array([])` cannot distinguish an empty character list from an empty numeric list. Both arrive as `[]`. An empty string `''` must produce `np.array([], dtype=np.uint32)`, not `np.array([], dtype=float64)`. The type information is lost before `to_array` sees the data.

This affects all character creation sites that could produce empty results: `Str.execute()` for `''`, `trim` on whitespace-only strings, `_fill_element`, environment defaults (`quad-DM`, `quad-LX`), and potentially any system function returning character data.

## Proposed next step

Before attempting the uint32 flip again, add domain error guards to all arithmetic methods (`add`, `subtract`, `multiply`, `divide`, `power`, etc.) in `numpy_array.py`. Currently they fall through from the `is_numeric_array` fast path to `_dyadic` which silently applies operations to character data. They should raise `DomainError` if either argument is a character array. This decouples arithmetic from the representation change and makes it safe to later change `is_numeric_array` to exclude uint32.
