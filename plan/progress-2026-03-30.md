### Pico Compatibility Fixes ‚Äî 8 regressions fixed

Previous session's API refactoring broke the Pico without updating `pico_eval.py` or checking MicroPython compatibility:

1. **`dataclasses`** ‚Äî `EvalResult` used `@dataclass`, replaced with plain class
2. **`_is_silent` import** ‚Äî moved to `Interpreter` method but `pico_eval.py` still imported from `repl.py`; updated to use `interp.execute()` API
3. **`importlib`** ‚Äî i-beam used `importlib.import_module`; replaced with `__import__()` + `getattr` path walking
4. **`os.path`** ‚Äî `OsFileSystem` used `os.path.exists/isfile/isdir/makedirs`; replaced with `os.stat` + bit masks and `os.mkdir` loop
5. **Bracket indexing on string literals** ‚Äî `‚ç¥'text'[idx]` parsed as `(‚ç¥'text')[idx]`; fixed `_item_string` in parser
6. **Bracket indexing on number vectors** ‚Äî `‚ç¥1 2 3[idx]` same issue; fixed `_item_number` in parser
7. **Bracket indexing after parens** ‚Äî `‚ç¥(expr)[idx]` parsed as `(‚ç¥expr)[idx]`; fixed paren reduction in `_build_items`
8. **Pico recursion test** ‚Äî skipped `test_recursion` (stack too small for non-TCO recursion)

## Lessons learned

- **Always check MicroPython compatibility** when changing code deployed to Pico ‚Äî no dataclasses, importlib, os.path, configparser
- **Playwright tests must verify visible output text**, not just DOM element existence ‚Äî tests that only check "entry exists" miss actual bugs
- **Kill background processes immediately** if they hang ‚Äî stuck test processes interfere with the user's server and subsequent testing
- **Bump version after every change** so the user can verify which code is running
