## Separate Executable (compute) from Reference (resolve) (v0.8.40 → v0.8.47)

Yesterday's A/B/C refactor gave Function/Operator/APLArray
first-class status but left `Executable.execute()` doing two
jobs: computing an APLArray (most nodes) and resolving to an
existing value (Var, AlphaAlpha, Nabla, Dfn, PrimitiveFunction,
SysFunc). The runtime `assert isinstance(result, APLArray)` in
`ctx.evaluate`, the `Var.execute -> APLArray` lie, and
`PrimitiveFunction(UnappliedFunction, Executable)` multi-inheritance
were all symptoms.

Eight commits split the concerns:

- **Step 1 (v0.8.40)**: Introduced `Reference(Executable)` with
  `resolve(ctx) -> APLValue` as a separate abstract. No callers yet.
- **Step 2 (v0.8.41)**: `Var` migrated — `execute` → `resolve`, return
  widened from the array-only lie to truthful `APLValue`.
  `Reference.execute` forwards to `resolve` so Reference keeps fitting
  wherever Executable is expected.
- **Step 3 (v0.8.42)**: `QualifiedVar`, `AlphaAlpha`, `OmegaOmega`,
  `Nabla`, `Dfn` migrated the same way.
- **Step 4 (v0.8.43)**: `PrimitiveFunction` and `SysFunc` — previously
  multi-inheriting `(UnappliedFunction, Executable)` just to sit in
  expression position — moved onto `Reference`; their `resolve()` is
  a trivial `return self`.
- **Step 5 docs (v0.8.44)**: First attempt at moving the apply/dop
  helpers off Executable onto Reference failed because
  `Applicable = Function | Executable` was also used for non-applicable
  rank/power operands. Reverted code; kept the discovery in docstrings.
- **Applicable split (v0.8.45)**: Introduced `OperatorOperand = Function
  | Executable` for operator-operand positions (which legitimately hold
  numeric rank specs); tightened `Applicable = Function | Reference` for
  the genuinely-function-applicable positions. `PowerDerived.right_operand`
  and `ForkDerived.f` kept `OperatorOperand` (both can hold a non-applicable
  Executable). Per-operator `derive_*` methods gained descriptive
  DomainError / SyntaxError raises via a `_require_applicable` helper so
  `+∘1` and `2⍤3` surface clear messages rather than bare AssertionErrors.
- **Helpers on Reference (v0.8.46)**: `_as_function`, `_as_operator`,
  `apply_to_*`, `apply_*`, `apply_*_dop` moved from Executable to
  Reference. `MonadicDopCall.op_name` and `DyadicDopCall.op_name`
  tightened to Reference.
- **Narrow execute (v0.8.47)**: `Executable.execute -> APLArray` finally
  enforced in the type system. `ctx.evaluate`'s runtime
  `assert isinstance(result, APLArray)` deleted. For the 5 call sites
  that genuinely need APLValue (dop operands, assignment RHS, fork-f
  resolution, apply_derived right operand) added `Executable.as_value(ctx)
  -> APLValue` which Reference overrides to call `resolve` directly,
  bypassing the array assertion.

## ∘ value-bind (v0.8.48 → v0.8.49)

`+∘1` was raising `DOMAIN ERROR`; in Dyalog this is currying
(`(+∘1) ω ≡ ω + 1`). Dyalog overloads `∘` on operand kind:

- `f∘g` (both functions) → compose (already supported)
- `f∘B` (right value) → right-bind `(f∘B) ω ≡ ω f B`
- `A∘g` (left value)  → left-bind  `(A∘g) ω ≡ A g ω`

New `BesideRightBound` and `BesideLeftBound` alongside existing
`BesideDerived`, with a 4-way dispatch in
`BesideConjunction.derive_dyadic`. Dyadic application of bound
forms raises `SyntaxError_` (matches Dyalog). All 7 expected
values cross-checked against running Dyalog via the MCP server.

v0.8.49 also replaced a bare `AssertionError` in parser's
`_as_evaluatable` with `SyntaxError_` — surfaces as a descriptive
error for `2∘3` and similar pre-existing parser gaps.

## Token / AST unification (v0.8.50 → v0.8.51)

The parser ran three parallel type systems — Token classes, CAT_*
integers, Node classes — translating between them. A `+` existed
three times: `FunctionToken` → CAT_VERB → `PrimitiveFunction`.

**First attempt (v0.8.50)**: per-type `Token` dataclass redesign.
Added 22 Token classes; raw isinstance count went UP from 168 to
215 because `.type ==` comparisons simply became isinstance calls.
User flagged it as worse — "much, much worse". Diagnosis session
identified the real problem: parallel hierarchies, not verbose
single hierarchy.

**Real fix (v0.8.51)**: value-producing tokens ARE the AST node
they'd become. Tokenizer emits `Num`, `Str`, `Var`,
`PrimitiveFunction`, `Omega`, `Alpha`, `AlphaAlpha`, `OmegaOmega`,
`Nabla`, `Zilde`, `SysVar`, `SysFunc`, `QualifiedVar` directly.
Only marker tokens (parens, delimiters, EOF) and `OperatorToken`
(axis-spec deferred to parser) remain as `Token` subclasses.

Parser collapse:
- `_parse_atom` 13 branches → 3.
- `_item_*` handlers mostly one-liners.
- `_classify_sysvar` tightens to `isinstance(tok, SysFunc)`.
- `_SYS_FUNCTIONS` moves from parser to tokenizer (it's a lexical
  concern).
- Token list type becomes `list[Token | Executable]`.

Net parser.py/tokenizer.py: −48 LOC despite ~290 lines of churn.
Raw isinstance count only moved by a handful, but the structural
simplification (flat dispatch, no translation step) is the win.

## Tokenizer reshape (v0.8.54 → v0.8.70)

Fourteen commits, sustained simplification push. Each step prompted
by user observation and implemented small. Collectively the file
went from ad-hoc char-walking loops to a clean layered design:
sentinel-padded source / one-shot regex readers / char-keyed
dispatcher.

- **v0.8.54 sentinels**: `self._source = list(source) + [None, None]`.
  `_current()` becomes a one-liner (no bounds check); `is not None`
  guards in equality / `in` checks removed where redundant. Predicates
  (`_isdigit`, `_isalpha`, `_isalnum`) accept `str | None` with the
  check moved inside once each.
- **v0.8.55–57 `_read_number`**: three progressive simplifications.
  First: scan forward then let int/float parse (no string accumulation
  in loop). Then: three linear phases (integer / fractional / exponent)
  replacing the has_dot/break state machine. Finally: regex-driven —
  `\d+(?:\.\d*)?(?:[eE][¯\-+]?\d+)?`. From ~20 lines to 7.
- **v0.8.58–60 high-minus validation**: `¯a` was producing an internal
  `ValueError` / `AssertionError`. Now validated at tokenize time with
  a clear `SYNTAX ERROR: High minus ¯ must be followed by a digit`.
  Assert-for-impossibility in `_read_number` replaced with `typing.cast`.
- **v0.8.61 `_read_string`**: `str.find("'", pos)` + slice + length
  check. Drive-by fix: unterminated strings now raise `SYNTAX ERROR`
  instead of silently returning partial contents.
- **v0.8.62 `_read_id` regex**: single pattern covers simple and
  qualified identifiers in one shot. No recursion, no sentinel `_pos+2`
  lookahead, no `::` assembly by hand. 15 lines → 5.
- **v0.8.63–64 `tokenize` loop**: redundant `while self._current() is
  not None` dropped; `if ch is None or ch == "⍝":` consolidates the
  two-break EOF/comment terminators.
- **v0.8.65 extract `_next_token`**: the big dispatch body moves into
  its own method returning the token; `tokenize` becomes a 9-line
  orchestrator.
- **v0.8.66 char-dispatcher**: `_HANDLERS` dict maps the seven
  char-specific branches (`'`, `⍞`, `⎕`, `¯`, `⍺`, `⍵`, `$`) to
  methods; each removed from `_next_token`'s if-chain. `⍺`/`⍵`
  removed from `SINGLE_CHAR_TOKENS` since their handlers disambiguate
  `⍺` vs `⍺⍺` (and `⍵` vs `⍵⍵`) internally.
- **v0.8.67 `if handler:`**: `.get()` returns None or function;
  functions are truthy; `is not None` unnecessary.
- **v0.8.68 `Token(Node)`**: Node's docstring already says "abstract
  base for all items on the parser stack" — tokens ARE that. Making
  Token a Node subclass collapses every `Token | Executable` annotation
  to `Node`. Dataclass-generated `__eq__` on each marker Token overrides
  the inherited Node.`__eq__` cleanly.
- **v0.8.69 strip quoted annotations**: forward-reference quoting was
  leftover habit; `Node`, `str`, `None` are all resolvable at annotation
  time. Only `'Tokenizer'` stays quoted (class-body self-reference).
- **v0.8.70 `_peek`**: `self._source[self._pos + 1]` →
  `self._peek()`; `_pos + 2` → `self._peek(2)`. Matches the parser's
  naming convention.

## Lessons learned / open questions

- Raw-count metrics can mislead. The v0.8.50 token redesign "reduced
  asserts" while adding more isinstance calls overall; only LOC and
  branch count showed the real effect. Watch for the right metric.
- Two false-positive "simplifications" in a row on parser.py
  (yesterday's apply_derived detour; today's first Token redesign).
  The pattern: touching a file you haven't fully diagnosed tends to
  make it worse. The successful run (v0.8.51 onward) followed an
  explicit diagnosis discussion.
- N-wise dyadic reduce divergence (`2+/1 2 3 4`) still unfixed; noted
  yesterday, still out of scope.
- Error-class precision (RANK vs DOMAIN for axis-out-of-range etc.)
  still worth a sweep.
